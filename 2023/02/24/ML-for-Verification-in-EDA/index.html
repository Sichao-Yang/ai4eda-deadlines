ML for Verification in EDA

ML for Verification现状综述

Author：Sichao Yang

Last update: 20230223

[toc]

1. 前言

近年来，AI（人工智能）特别是其领域内的ML（机器学习）和DL（深度学习）技术，已在诸多应用领域取得了突破。在传统的半导体领域，研究者们也探索了基于机器学习的芯片设计新方法。这些新算法通常会反映在EDA（芯片设计自动化工具）上。因此，这一研究方向通常可以被称为【ML for EDA】，业界针对【ml for EDA】的介绍和综述文章已经有很多了1234，但还没有专门针对【ML for Verification】的文章，本文希望能弥补这样的空白，帮助读者梳理该研究方向上的历史，现状及发展。

我们首先会简单介绍Verification in EDA（EDA里的验证工作）和ML的基本知识，然后说明为什么需要研究【ML for verification】，接着会后介绍该领域的发展趋势及各细分方向上的代表性研究工作，最后提出该领域当前面临的挑战及未来展望。

2. 背景介绍

2.1. 什么是verification in EDA

要理解Verification in EDA，首先让我们看看芯片研发流程，以数字芯片为例：

  1. 输入系统规格需求，系统设计团队首先会输出架构设计和模块功能设计文档，
  2. 然后由前端设计团队进行逻辑设计形成RTL代码，经由逻辑综合logic synthesis形成设计电路的technology dependent的网表，
  3. 再交由后端设计团队进行规划，布局，布线，时钟树综合等步骤，形成.gts文件，交由代工厂进行芯片样本生产制造（流片），
  4. 上述的设计流程里的每一步又会伴随着各种验证工作：在制造之前的 pre-silicon 阶段，前端设计后会进行功能性验证（动态仿真，静态检查等），后端设计后的物理验证（Static-Timing-Analysis，Power-Analysis，Design-Rule-Checking，Layout-Versus-Schematic-Checking，Clock-Domain-Crossing-Checking等等），只有sign-off checklist上的所有检查都满足了设计才能被流片。在制造之后的 post-silicon 阶段，还会对真正的样品进行各种测试来进一步检查芯片质量，性能等指标。

在这个过程中，工程师既需要1. 权衡功耗频率面积 （PPA）等多个设计目标，还要2. 确保制造出的芯片将正确运行各种功能，并且要3. 最大化设计效率（直接影响产品上市时间）。这么多复杂的约束和目标意味着现代的芯片研发必须借助强大的工具才能实现，而这些工具就统称为EDA。

如上所述，在芯片研发流程里一个非常重要的工作就是验证芯片的各项指标是否满足预期。由于芯片的研发成本非常之高，验证的重要性不言而喻。有研究表明，验证在某些项目中的耗时占比甚至超过70%8。

2.2. 什么是ML

近10年来，伴随着ML技术在图像识别、自然语言处理、搜索推荐等领域的突破性进展，ML也越来越多地被引入到各种领域，例如AI for science，AI for Health-Care，智能制造，安防，教育，交易，供应链，自动驾驶，游戏等等。可以说只要这个领域能产生数据就有应用ML，统计分析或数据挖掘的潜力。在EDA领域里，学界和工业界同样在不断的投入资源研究ML相关的技术及对应的应用落地。

那么什么是ML呢？简单来说，ML就是在历史数据中发掘规律然后在当前数据中识别规律的技术。理论上这个识别规律的模型可以用来拟合任意复杂的从输入到输出的映射关系。

从学习（发掘规律）方式上来对ML分类，可以大体分为有监督，无监督和强化学习。其中有监督学习需要构造样本标签，无监督学习则不需要。通常只要是能构造出明确的标签，模型的效果会更好，但标签的构造往往需要花费大量的人力和时间成本。不同于前两者，强化学习的数据来源需要智能体主动和环境互动来产生，目标是学习出能根据当前环境给出最优决策的策略函数。它特别适用于目标奖励距离当前状态遥远，需要多步决策达成的任务，例如下棋。

ML的流程可以分为训练和推理两个阶段，在训练的部分我们需要通过历史的数据来运行算法获得对应模型权重的数据保存起来，然后在推理阶段根据新的数据和模型的权重给出推理结果。

具体来说，ML包含了下列几类常见的功能：

  1. 预测 - 依据观测变量ML来估计目标变量，例如后端设计中的功耗，延时，阻塞预测。这类模型可以在设计初期给出一些快速的性能预测，来筛选掉一些不好的设计。EDA的传统算法里包含了很多类似的预测函数，ML可以根据历史数据给出更准确的近似函数。
  2. 优化 - 根据问题结构，ML在可行的输入空间里搜索出目标值最优的坐标，例如前端设计中通过逻辑综合来优化电路网络，后端设计中通过布局布线来优化线长，chip area等。ML可以集成在传统的优化算法里，利用历史或当前每一次迭代产生的数据进行学习，为传统的优化算法提供更快速的evaluation function，或者根据每次迭代的结果给出更准确的指导优化方向实现收敛。

  3. 决策 - 给定系统当前状态和目标，ML来决策下一步的行动，例如前端设计中存在在多个子问题求解引擎里选择最合适的场景，例如门级电路综合，形式化验证。后端设计中的布局布线也可以被看成一个决策的场景，给定当前的布局，ML会自动选择下一个该放置的macro以及其对应的location。这些场景都可以尝试通过ML来学习历史数据并做出比传统方法更优的决策。

  4. 生成 - 依据学习到的能表达数据分布的核心特征（隐空间变量）ML可以生成新的符合样本分布的数据点，例如后端设计里的layout pattern generation for DFM6，前端设计里依据人类语言的spec生成verilog代码7等。这类模型可以非常短的时间（对应的往往是真实数据的收集代价非常高）生成多样性丰富的数据，生成的数据可以定向的输出给下游的任务（或许是另一个ML学习器）。

  5. 异常检测 - 通过学习历史正常数据的分布，ML可以判断新的数据是否来自于同一分布，例如在后端设计里布局后检测版图上各种violation的hotspots，验证里检测仿真运行日志中存在的异常事件等。智能的自动化异常检测技术可以在产生海量数据的系统运行监控场景下节约大量人力。

2.3 为什么研究ML for verification

芯片的规模（Gates/Chip）作为衡量芯片性能的重要指标，根据摩尔定律一直保持着每周~1%的提升，当下大规模的芯片已经可以包含超过 100 亿个晶体管，而芯片研发的生产效率（Gates/Day）却没有办法保持同步的提升，两者之间存在一定距离，这被定义为productivity gap问题9。这一问题在可预见的未来，会随着系统复杂度的提升和材料物理边界的逼近，被进一步放大。现有EDA工具需要不断创新，来提升设计和验证效率。其中一个重要的创新技术点，就是引入数据驱动的方法10来改进现有的算法和流程，而针对验证的ML技术即为【ml for verification】。

具体来说，在芯片研发的过程中，

  1. 遇到的问题，通常会面临着近似unbounded的搜索空间，所以为了让问题的求解变的tractable，传统的decision making需要依赖于大量基于经验的手工制作的heuristics启发式策略，但求解的速度依然逐渐成为瓶颈。

  2. 由于问题的复杂性，通常会产生大量的数据（配置参数表，源码版本，更新日志，网表信息，运行日志，验证报告，波形输出等），传统的算法通常没有有效利用这些数据。

以上两个特性都很好的匹配了ML技术，通过引入ML可以学习大量历史数据形成经验知识，再结合现有的技术可以进一步加速问题的求解，辅助研发人员做出更优的决策。

3. 学界工作

3.1. 概要

4.1.1 趋势

如综述11所述，在5类EDA相关的顶级国际会议上（TCAD, ICCAD, DAC, ASP-DAC, ESWeek），对2016-2020年的非邀请类文章做了统计，首先EDA方向的论文整体数量在逐年增加，其中和ML相关的文章占比逐年增加，从2016年的5%已上升至2020年的11%。

这篇综述里对ML的文章做了分类（图2）分别为：Verification/Test，Manufacturing，Physical Design，Logic Synthesis，System-Level Design-Space-Exploration & High-Level-Synthesis。可以看到的一点是，后端上的研究比例近几年增加明显。Physical design + manufacturing的比例在20年占比超过了65%。20年至今可以看到的是后端的研究占比在进一步增加，其中主要原因笔者认为是：后端的设计和制造涉及到的数据类型天然的可以被表示为几何图形，非常适合把原本应用在图像上的ML和DL技术做横向的迁移。而前端设计以及验证里的问题，通常都是基于布尔函数的组合搜索问题，这一类问题是目前公认ML或者DL很难求解或者精度上还没有能超越传统方法的问题。

从图3上可以看出，深度学习里以神经网络（NN）为代表的技术占比在近年明显增加，其中主要是图神经网络GNN，生成对抗网络GAN和深度强化学习RDL。即便几年前用传统ML技术（支持向量机SVM，逻辑回归Logistic regression）解决的问题，也随着DL的兴起正逐步尝试过渡到了DL技术上来提升精度，泛化性和统一建模范式。

3.1.2 分类

验证的目的简单来说就是首先在有限时间内，尽可能全的覆盖到所有的系统状态的同时尽可能多的找出bug；然后尽量高效的寻找这些bug对应的原因，并进行修复以确保系统的正确性。它可以依据其实现的方式分类为仿真及形式验证。在仿真验证里又可细分为：为了提升测试效率和调试效率进行的ML研究工作；在形式验证里可分为：提升SAT求解器求解效率及模型检测效率上的工作。这几类问题可以被描述为：

  • test efficiency - 为了高效的覆盖到所有系统状态 coverage closure，我们希望在所有的test中有尽量少的重复测试（既减少不会带来新增覆盖率的测试）。同时，为了找出bug，通常需要在不同位置通过插入assertion的方式来检查期望满足的系统不变性。而当面对复杂系统时，推导和构造assertion本身就是一件困难的工作，所以如何智能的生成assertion并检查现有assertion的质量，就是一个提升测试效率的方向。
  • debug efficiency - 芯片设计的每一轮迭代伴随着回归测试，每一次回归测试后都可能产生成百上千的错例。工程师在预诊的时候需要对这些错例进行初步的诊断和归类再交给对应的专家进行细节调试。一些直觉的问题是：预诊时怎么把它们归类分组，把他们分派给哪一只团队进行debug；根因分析的时候是从哪一个错例入手，通过什么数据去分析。很多情况下这些问题的答案，都来自于工程师的经验判断。利用历史数据，ML技术有潜力在这些环节上提供帮助。
  • model checking - 芯片电路可以被看做一个巨大的有限状态系统，检查这个系统就是在检查在给定的输入状态后系统根据预定规则游走在状态空间里时是否会违反用户给出的各类specification。检查的算法有很多种，他们各自适合不同结构的电路，ML可以利用历史数据更智能的匹配算法和电路更高效的完成检查。
  • SAT （Boolean satisfiability） solver - 在Model checking的框架下，通常在底层会调用SAT。SAT的求解本身是一个NP-complete问题，会涉及到tree search，在传统的搜索里存在很多针对不同类型节点做branching的heuristics，目的是更高效的完成搜索。利用历史数据，ML模型可以更好的对搜索方向给出建议。

根据上面的分类，下图给出了2003年~2023年过去20年较为重要的【ML for verification】的文章清单，以时间排序，同时给出了各自对应的ML算法。其中标红的论文，会在下一节选作介绍。

3.2. 具体介绍

3.2.1 测试效率

上节test efficiency板块标红的4篇论文均出自Shobha Vasudevan的团队，她是University of Illinois的教授，同时是GoldMine（10年开发，后成功被多家全球领先的EDA公司购买license）的作者，还长期与Google Research合作。她的核心研究思路是寻找能更好结合static anlaysis，simulation，formal和ML的框架来提高测试效率。

作者团队10年设计的GoldMine12就是一个典型的例子，算法希望能通过data mining在验证初期智能的生成更多的assertion来节约manual effort。算法整体flow如下图所示：首先给定一个需要挖掘的目标的输出变量，通过data generator我们可以获得很多测随机输入下的仿真traces构造训练集，然后A-Miner会对这些数据进行分析生成可能的不变规则的集合，这些规则会被输入到formal verification来筛查，通过的就会被转化成rtl assertion自动的加入代码里。另外，static analysis模块会提供额外的信息（cone of influence，variables’ topological ordering等）来提升A-miner精度。Assertion在被添加到代码里之前，也可以选择通过A-val模块提交给验证人员做核查。

通过论文中的实验显示，原始directed test拥有76%的coverage，而通过添加GoldMine生成的assertion，coverage能提高至100%。

虽然上例中的Goldmine能自动生成assertion，但是input vector还是随机生成的，会存在很多相似或低效的测试，coverage closure的运行效率本身并没有被提高。为了解决这个问题，作者又分别在11年和12年提出了两种智能生成input vector1314的方法，两种都是利用静态分析给出的counter example，其中12年的方法使用了结合GoldMine的方式：原始的GoldMine里只用到了formal检查成功的assertions。而如果失败了我们会获得反例，对这些这些反例进行仿真，系统可以到达之前没有触达的状态，产生的traces也可以被继续挖掘产生assertion，反复迭代最终coverage会达到100%。另外，论文里还用incremental tree替代了原始的decision tree来进一步提高学习的效率。

作者通过实验证明，该算法可以在不给定任何定向测试案例的情况下，能自动的高效实现100%覆盖率的收敛。

从2010之后的十多年时间里，ML里基于神经网络的DL技术得到了突破性的发展，所以作者在21年发表了基于Graph NN15的算法模型，期望能找到一种更通用的表达电路结构的方式，来降低之前提出的方法里构造特征时对domain knowledge的依赖以及模型难以从当前问题泛化到其他问题的局限性。其基本思路为：通过对RTL代码进行解析，转化其为Control Flow Graph结构。在下图1的例子里，每一个节点就是一行代码，每一个always里包含了一个局部的子图。用图网络学习器，我们可以在图上学习 is_hit = f(input vector, cover point vector) 的映射关系。cover point vector代表系统里所有需要覆盖的点，每一个点的取值为 {0, 1}。input vector就是每一次测试时的输入变量赋值。is_hit是指给定input vector和目标第i个cover point（cover point vertor[i] = 1 and cover point vertor[k] = 0），系统是否能触达。

具体来说，在模型训练阶段：

  1. 首先是针对下图2左边的Cover points进行embedding：

    • 对每个node包含的代码的每个word进行tokenization，通过word2vec转化成向量，然后作为一个序列输入LSTM网络，输出为code embedding。再concate上当前节点的attributes（节点类型，fan-in\out数等），形成node feature；
    • 然后通过GCN的feature propagation方法，对图上的每一个节点，把相邻节点上的feature信息传播到当前节点来更新信息；
    • 再接着通过输入的cover point vector对所有node做masking，只留下目标的cover point子集对应的节点feature；
    • 最后通过LSTM把这些节点的feature fusion到一起，形成CoverPoint embedding向量；

  2. 同时针对test input vector（下图2右侧）生成embedding：把它通过一个feed forward神经网络形成向量；

  3. 最后把1和2生成的向量concatenate到一起，输入分类器得到is_hit结果。

模型推理的阶段，就是给定目标要cover的test point，不断搜索test vector，直到输出的is hit为真。

作者也在实验中显示，利用50%的coverage实验数据，GNN的算法在输入vector的预测任务上可以稳定的overperform其他基线算法，并且在小问题上训练的模型具有一定针对新问题的迁移能力。

除了上述4篇来自Shobha Vasudevan的论文，Intel团队的Cliff和Item工具20里也是利用ML技术来提高覆盖率收敛，例如他们首先从历史测试中检测目标变量（covering point）被激活数明显少于均值的测试，并以激活它为目标去挖掘与之相关性高的输入变量，并生成新的测试的输入。还有从官方宣传上看在所有历史项目上平均提高至少20%测试效率。

另外，除了上述特征，研究者还通过dump错误处附近的stack信息27，比较state信息和ISS（instruction-set-simulator）差别等方式来构造特征。

3.2.2 调试效率

上节debug efficiency板块标红的3篇论文171819均出自Andreas Veneris的团队，他是University of Toronto的教授，同时是Vennsa Technologies公司的创始人（第一家专注于研发自动化功能性验证调试工具的eda公司）。他的核心研究思路是通过结合formal verification和ML的方法来对回归测试里的错误案例进行智能预诊及错例分析。

回归测试后，通常的流程时会对错例初步的诊断并按此进行分箱，然后按照分箱后的结果分发给不同的debug人员进行更细节的调试和修正，如下图所示16。这个流程通常凭验证人员经验完成，如果分箱错误或者分发错误，就会造成大量的返工消耗项目时间。ML技术的引入是希望通过挖掘历史的数据来提高预诊精度和整体效率。

在14年，作者在论文17里提出了一种利用聚类进行分箱的方法。为了对错误集合 进行预诊分箱，针对每一个错误需要构造其独特的特征向量，然后就能通过ML聚类算法对这些向量进行预诊分箱了。具体来说，算法的流程为：

  1. 特征构造：
    1. 给定 Fi，通过formal工具获得suspect set ，这里面的每一个s都是从Fi的cone of influence（所有和输出错误相关的输入变量）集合里筛选出来的能通过强制改变其取值纠正错误 Fi 的变量。
    2. 对所有Si取Union，我们得到一个超集SU： and 。针对Fi，他的特征向量为，如果，则取值为1，否则为0。
    3. 接着还需要对 里的每一个取值为1的suspect进行加权，加权的思路是如果该suspect对error影响大，则权重应该设置的较高。例如，如果该suspect距离发生error的时间点较近则权重应该较高；同时作者argue如果从触发suspect到error发生这段区间里，suspect的toggle rate不高就说明触发他和产生error这两个event的因果相关性比较高。

作者通过实验证明，自动化的预诊分箱，可以达到89%的精度，而分箱执行时间对比人工方式可以忽略不记。

  2. commit logs，通过解析，算法可以判断该次commit是属于bug fix还是function add，如果是function add则该版本对应的suspect的权重会增加。

对比上一个算法和人工方式，该模型精度在实验中表现进一步提升。

前两种方法的性能瓶颈都在利用formal工具生成S集合上，因为这个过程耗时占比最长（大于80%）并且是做clustering的前置条件。为了解决这个问题，作者在20年19提出了在formal工具搜索S集合的过程中利用ML模型提前预测S的方法，这样可以提前提供高rank的suspect给后续的分析或者clustering模块。

这里作者提出并对比了两种模型，一种是probabilistic graph model，一种是深度学习里的auto-encoder model。第一种模型的原理是说如果一个suspect si被另一个suspect sj dominate（也就是说每一条通过 si 到达输出的路径上都必须经过 sj ）那也意味着如果si的改变能纠正错误，sj的改变一定也能纠正错误。即： 。作者把这个概念延申到了集合dominance和概率dominance上。并且通过概率图来进行建模，这样就可以利用模型来输出 的概率。也就是说，给定formal求解器当前搜索到的S集合，概率图可以预测sj是否为嫌疑犯的概率。如下图例子：给定已观测到的S={s1, s2}，根据他们的dominance关系图，我们可以预测s3，s4，s5，s6各自是否为suspect的概率。

第二种模型，作者在典型的auto-encoder基础上做了一定的修改，来保证模型可以适应输入为不同cardinality的S集合。作者在实验中对比之前的方法，通过引入预测模型，嫌疑犯suspect的搜索效率在初期得到了巨大的提升。auto-encoder的结构虽然比概率图预测精度要高，但是对数据的鲁棒性更低需要的数据也更多。

另外，由于debug中涉及到对大量信号，不同类型数据的检查，怎样对处理数据使之更适合工程师观察（data visualization）也是一个潜在的研究方向，论文28里有很多相关的工作。

3.2.3 SAT 求解器

Satisfiability solver的是形式验证里非常重要的方法，他的求解方式通常会涉及到对树结构的搜索，搜索中会用到很多heuristic策略，ML在这上面的结合也有很多研究。大体来说可以分为两种，一种是通过数据学习选择器来选择不同branching的方法或者是不同的求解器；另一种是end-to-end的方案，直接学习出问题的结构并且进行sat求解。

3.2.3.1 启发式选择

Heuristic selection - SAT求解时，算法在每一次branching的时候都需要选择一个自由变量，有很多heuristic的选择策略，在论文23中，作者利用强化学习的建模方式把DPLL的branching选择问题建模成一个通过连续选择分支策略实现利用最小运算量找到SAT目标的问题。强化学习需要定义action，state，transition map和reward。这里的action是对7种不同的分支策略选择，state应该和问题的结构以及当前状态相关，很遗憾作者通过实验发现当前自由变量数量对学习有正面作用并以此作为state，transition就是给定s选择了策略a后会关联的下一个状态，reward是选择action后当前的计算的cost加上后续branching的cost。

在作者的实验中，针对4个不同问题，给定足够数据，ML的branching选择器，可以做到略微优于当前问题最优单一策略的效果。

不同于选择branching，在SATzilla0722里，作者尝试把选择求解器的问题交给ML模型：根据每一类待求解电路的（per-instance base）特征来选择最适合他的solver(s)（例如可能的求解速度最快的4个solver）来进行求解。通过专家经验，作者构造了48种特征。在2007 SAT Competition中，在SAT & UNSAT等track上都获得了冠军。

一样的问题，在17年被作者29extend到了CDCL算法的SAT solver上，利用基因编程（GP）的技术能在runtime的时候对一类问题自动的优化variable scoring heuristic，进而取得更快的收敛速度。

3.2.3.2 端到端

不同于结合传统求解器的方式，论文25里的作者提出了end-to-end的方法，以期望获得比上面更好的全局最优解。他们尝试直接通过GNN的建模方式来预测布尔表达式是否SAT。之所以选择GNN，是因为对比把CNF转化成顺序的clause的序列再用RNN建模，它能保证permutation invariance和negation invariance。这里使用的GNN，实际上就是一个bipartite graph，特征信息在literal和clause之间交替传递。每个literal对应的特征向量可以是一个one-hot的encoding vector。

2019年同年，论文24的作者提出了把电路转化成有向图DAG的形式用正反向两个RNN来建模，并用一个近似的evaluator来对模型给出的counter sample进行打分的结构，对比论文25取得了性能上一定的提高，但从两篇论文的实验上看距离传统SAT的精度还有很大差距。另外，他们的数据集构造也包含了比较特定的假设。

3.2.4 模型检测

Model checking需要检查设计的系统代表的state space是否存在有违背用户给定specification的错误。一种常规的检查方式是给定输入initial state，根据定义的transition rule在state space里walk，算法需要检查这些path是否符合条件，同时有策略的提供input来激发错例。但复杂系统存在很多的state，即使有传统的策略，依然存在state space explosion问题。

所以作者在论文里26提出了基于数据瓦挖掘的策略学习方法：首先遍历一小块儿的state space，然后筛选遍历后获取的path traces数据，保留所有的counterexample traces，再从里面挖掘出frequent sequences或者是构建一个probabilistic graph，最后利用这样的知识作为策略，指导后续更大空间里的高效搜索。

从实验上看，对比开源工具里的其他算法，数据挖掘的方法能得到更快更准确的结果。作者提到制造训练数据本身可能是一个耗时的工作。

4. 商业落地

在工业界，一些EDA 公司和芯片设计公司对 【ML for verification】 的方向表达了很大兴趣。三大 EDA 公司 Cadence，Synopsys，Siemens EDA也都推出了一些融合了 ML 算法的新产品。华为，intel，微软，IBM，谷歌和英伟达的芯片团队也在积极探索该方向的商业落地性。

具体来说，Synopsys已经将ML技术应用在了验证流程的各个环节中，包括静态验证 (VC SpyGlass)，形式验证 (VC Formal)，仿真加速 (VCS DPO)，覆盖率收敛 (VCS ICO)和调试效率提升 (Verdi RDA）中，分别可以达到各环节上2~10倍效率的提升21。Siemens EDA在它的 Questa 里集成了ML技术 （Questa Verification IQ）。Cadence也构建了AI-driven的验证平台 Verisium。

5. 挑战

经过多年的研究【ML for verification】得到了极大的发展，但也依然面临的一些开放的挑战，简单总结如下：

  1. 实际落地 -

    1. EDA行业的流程标准都是经历了几十年的发展，相对完善，当前ML技术通常是作为一个传统功能模块的替代，而一个end2end的ML方案很难被接受。
    2. 开发和维护 ML 模型以及与之对应的数据链路和服务架构，需要大量 ML 背景的工程师，而这对中小型EDA公司来说并不容易。
    3. ML模型的引入可能会导致数据安全和模型所属权的问题，比如说客户A的设计和供应商的IP可能会在不知情的情况下被ML模型学习到，再在推理的场景下被客户B使用。
    4. 同时IC领域的发展迅速，每一代technology node对应的设计与制程都不一样，当前训练的ML模型很可能有效期太短。

    ML 方法如何更好地融入现有的 EDA 工具与芯片研发流程。这一点也许需要研究者与工业界共同进行探索。

  2. 数据有限 - 训练所需的数据通常非常难以获得，也导致模型的泛化性问题更为显著。对于这个问题，一方面一些研究者开始提供公共的，规模更大，多样性更好的EDA相关数据集；另一方面研究者探索了在保护数据隐私条件下的进行模型训练，例如联邦学习。

  3. 可解释性不好 - 现在的ML技术很难给到用户其预测结果的合理解释，使得用户对于其预测结果的信任度不高，特别是在IC研发这样安全性要求非常高的领域，这一问题会进一步被放大。

  4. 组合优化（CO）问题 - 很多的EDA算法都是在求解组合优化问题，而ML在这一方面并没有取得类似于图像和语言上的成功。ML没法保证optimality，甚至寻找可行解也对ML来说不容易（不可微分，约束难描述）；还没有成熟的网络结构对应CO问题；由于组合空间太大，ML模型泛化性问题也更难解决；同时CO里，类似样本（相似电路）的求解并不是频繁发生，所以数据的收集也是挑战。

为了解决上述的问题，未来ML的技术4需要朝着reusability + evolvability + Interpretability 的三大方向继续发展：

  1. Reusability - pretrain+finetune范式的引入，可以用统一预训练的大模型加对应不同任务的少量数据微调的方式来提高模型效率和泛化性。这里面比较有潜力的方向：adaptive modeling，transfer learning 和 bayesian learning。
  2. Evolvability - 模型需要不断适应变化的环境以解决data, concept drift问题，里面包含：incremental learning，reinforcement learning，active learning。
  3. Interpretability - 模型需要有更好的解释性，比如 knowledge distillation，interpretable ML，trusted ML等。

同时，为了更好的支撑ML技术在verification里的应用，我们应该倡导发展出更对ML更友好的基础设施，例如：生成更透明，包含更多关系的数据，构建更适合于ML数据使用的数据仓库，设计更适合ML模型和EDA系统交互的接口20等。

6. 结语

本文介绍了【ML for verification】这一研究方向，期望起到抛砖引玉，启发业界同仁的作用。可以看到的是，ML技术在 EDA Verification 领域提供的优势是极其独特的，相信未来会有更多的 ML 技术通过EDA验证工具融入到芯片研发流程之中。未来 ML 技术究竟会在流程中占到多大比重，取决于大家共同的探索与实践。笔者期待 ML 技术能够推动 EDA 验证产业，特别是国产 EDA 验证工具的发展。
